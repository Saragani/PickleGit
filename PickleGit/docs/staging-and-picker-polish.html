Staging & Picker Polish

User testing against a real Bitbucket repo turned up a real bug the same day the PR dialog shipped: the "Into" branch picker rendered as a plain white Windows dropdown, not the app's dark theme. Fixing it properly meant re-templating ComboBox from scratch — which also made room for the type-to-filter search the branch list actually needed. Alongside it, three long-standing Local Changes gaps close: multi-select now really works for stage, unstage, and discard, and there's finally an Unstage All to match Stage All.

The ComboBox Bug

PickleGit had never used a ComboBox anywhere before the pull-request dialog. The theme's existing style only set Background/Foreground/ BorderBrush — which changes the closed box a little, but does nothing for the dropdown popup, because the default Aero template's popup chrome is hard-coded to system colors regardless of those three properties. The result, caught by hand-testing against a real repo: a dark app with a stark white branch list.

After
Master▾
  • Fully re-templated, not patched — ComboBox and ComboBoxItem both get complete dark ControlTemplates in DarkTheme.xaml, the same treatment already given to ScrollBar, ContextMenu, and MenuItem — an unstyled popup wasn't a one-off, it was the last stock WPF control left untouched.
  • Type to filter — the box is now editable (IsEditable="True") with WPF's built-in single-character jump-to-match turned off (IsTextSearchEnabled="False") in favor of hand-rolled substring filtering on every keystroke — useful the moment a repo has more than a handful of remote branches.
  • The arrow, not the whole box, opens it — the first pass at the template put an invisible ToggleButton across the entire control, which is how the default non-editable ComboBox works — but on an editable one that also blocks every click meant to place a text caret. Caught in review before shipping; the toggle now only covers the narrow arrow strip.

Multi-Select Everywhere

The commit list has supported multi-select since early on, via ListViewMultiSelectBehavior syncing ListView.SelectedItems to a view-model collection. The two Local Changes file lists never got the same treatment — you could Ctrl-click three files, but Stage / Unstage / Discard only ever acted on whichever single file the click or right-click happened to land on.

How the target list is resolved

  1. Both file lists now run SelectionMode="Extended" with their own ListViewMultiSelectBehavior, feeding SelectedWorkingFiles and SelectedStagedFiles — the same pattern SelectedNodes already uses for commits.
  2. A per-row +/−/✕ button click doesn't change ListView selection at all — clicking a button never selects its row. So ResolveFileTargets takes both the clicked file and the current selection: if there's a multi-selection and the clicked file is part of it (or nothing specific was clicked, e.g. a keyboard/menu invocation), the whole selection is the target; otherwise it's just the one file that was clicked.
  3. Right-clicking already behaves correctly for free — WPF's own ListBoxItem only replaces the selection on right-click when the clicked row wasn't already part of it, so a context-menu "Discard changes" on a 3-file selection reaches all three without any extra code.
  4. GitService.StageFiles / UnstageFiles / DiscardFiles batch the whole target list into one Commands.Stage/Unstage/ CheckoutPaths call rather than looping single-file calls — one index write instead of N.
✅

Discard confirmation scales with the batch

Discarding one file still asks "Discard changes to 'Foo.cs'?" — discarding five now asks "Discard changes to 5 files?" instead of firing the same confirm dialog five times in a row.

03

Unstage All

Stage All has existed since early on, sitting in the Staged Files header — the panel whose contents it fills, not the one it reads from. Unstage All follows the same logic and lands in the same header, right next to it, rather than in the Unstaged Files section it would seem to belong to at first glance: both buttons control what ends up in the Staged list.

  • GitService.UnstageAll() — a single Commands.Unstage(_repo, "*") call, the mirror image of the existing StageAll()'s Commands.Stage(_repo, "*").
  • Disabled with nothing to do — the button's CanExecute checks StagedFiles.Count > 0, the same guard style already used for the commit button, so it's visibly inert rather than a no-op click when the staging area is empty.

What's left now

All five items from this round's punch list are in.

  • DONEDark "Into" picker — full ComboBox/ ComboBoxItem re-template, matching every other themed control in the app.
  • DONEType-to-filter branch search — substring match, live as you type, no more scrolling a long branch list by eye.
  • DONEDiscard on the full selection — batched into one CheckoutPaths call, one confirm dialog.
  • DONEUnstage All — next to Stage All in the Staged Files header.
  • DONEStage / Unstage on the full selection — same ResolveFileTargets logic backs both directions.
