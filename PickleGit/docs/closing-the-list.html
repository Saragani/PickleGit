01

Diff & History

Four of Phase 03's five deferred items, all in service of the same idea: the diff panel should tell you more than a wall of plus and minus signs, and it should be possible to ask a file what happened to it before now, not just in the one commit you happen to have selected.

  • File history & blame — right-click any file for a new FileHistoryWindow — a commit list scoped to just that path via repo.Commits.QueryBy(path), with a diff preview per commit, and a Blame tab reading repo.Blame() to annotate every line of the current file with the commit, author, and date that last touched it.
  • Image diff — png / jpg / gif / bmp / ico changes now render as an old-vs-new side-by-side preview instead of a binary-diff dead end, reading blob bytes straight from libgit2 for the old side and the working tree or index for the new.
  • Side-by-side diff — a toggle next to the existing unified view aligns old and new text in two columns, pairing consecutive deleted/added line runs the same way the word-diff pass already did, with filler rows where a line was purely an addition or a deletion.
  • Syntax highlighting — a hand-rolled, dependency-free lexer (not AvalonEdit — it would've broken the flattened DiffItem virtualization every diff view depends on) covers C#-family languages, Python, PowerShell, CSS, XML/XAML/HTML, JSON, and Markdown, layered underneath the existing word-diff highlight rather than replacing it.

Two highlight passes, one line of text

  1. Word-diff spans (background tint) and syntax spans (foreground color) are computed independently and stored on the same DiffLine.
  2. The render-time behavior merges both span sets into one sorted list of breakpoints, so a single Run can carry a syntax color and sit inside a word-diff highlight at the same time.
  3. Where neither pass has an opinion about a stretch of text, the Run is left with no explicit color at all — it just inherits the line's existing added/deleted/context foreground, so an unrecognized file extension renders exactly as it did before this update.
🐞

Caught in testing

The image-diff panel's "file added" / "file deleted" placeholder text was showing up underneath both images even when both sides had loaded successfully — a red square and a blue square, with "no previous version" and "no new version" captioned across them regardless. The visibility binding used NullToVis (visible when the value is not null) where it needed InvertNullToVis (visible when it is) — the two converters exist specifically because both directions come up constantly in this codebase, and it's easy to reach for the wrong one on a fresh binding. Caught immediately on the first live screenshot, fixed by swapping the converter.

02

Settings & Shortcuts

Phase 05 shipped a settings window with two sections. This round took it the rest of the way: a real identity system, a UI section, and every keyboard shortcut in the app made editable instead of fixed at compile time.

  • Identity, resolved in order — a new Profile section holds a default name/email plus a per-repository override; opening a repo now fills in the author fields by checking, in order, the saved override for that path, then the repo's own user.name/user.email config, then the default — the same fallback chain a real git identity setup follows.
  • UI preferences — a configurable date format (a plain .NET format string, applied through a converter with a static in-memory cache so it costs nothing per row render) and a toggle to skip the discard-changes confirmation for anyone who's decided they trust themselves.
  • Rebindable shortcuts — all 13 shortcuts — refresh, push, pull, fetch, tab navigation, hunk navigation, the command palette itself — now live in one ShortcutManager list instead of a static block in MainWindow.xaml. Rebinding one calls straight into MainWindow.RebuildInputBindings(), so a new gesture works immediately, no restart.
🐞

Caught in testing

The first pass at the settings layout used a real WPF TabControl with a custom TabItem template. On screen, every tab header was clipped mid-word — "Gene", "Prof", "Integratic" — despite the window being wide enough by a comfortable margin. Rather than chase the default template's header-panel sizing further, the tab strip was replaced with the same button-row-plus-swapped-panel pattern already proven elsewhere in the app (the diff view's Unified/Side-by-side toggle, the file-history window's History/Blame toggle) — six buttons, six named panels, one Click handler that shows one and collapses the rest. Fully legible on the first retry, and one fewer bespoke control template to maintain.

03

Interactive Rebase

The one item deferred as too large for Phase 04. Pick, reword, squash, fixup, and drop commits before a rebase runs, reordered with simple up/down controls rather than a drag gesture — and all of it without ever handing control to an external text editor.

  • The todo list — a new dialog lists every commit between the chosen base and HEAD, oldest first — the same order git itself applies them in — each with an action that cycles Pick → Reword → Squash → Fixup → Drop, and up/down buttons to reorder.
  • Reword without an editor — git's own reword todo verb would launch $GIT_EDITOR mid-sequence — exactly the interactive stop this app exists to avoid. A reworded commit is instead written as pick followed by exec git commit --amend -F <msgfile>, where the new message was already typed into the dialog and written to a temp file before the rebase ever starts.
  • Conflict banner, for free — interactive rebase uses the same rebase-merge directory as plain rebase, which the Phase 02 conflict banner already reads for step N of M — an interactive rebase that stops mid-sequence shows up in the existing banner with zero new wiring.
🛑

Caught in testing — the whole feature was dead on arrival

The mechanism above relies on GIT_SEQUENCE_EDITOR pointing at something that copies a prepared todo file over the one git generates. The first implementation pointed it at a generated .cmd batch file, quoted the way an ordinary Windows command-line argument is quoted. It built cleanly, and it failed every single time it ran, with:

What actually happens on Windows

  1. Git for Windows does not run GIT_SEQUENCE_EDITOR or an exec todo-line through cmd.exe — it runs them through its own bundled MSYS sh.
  2. A bare, unquoted Windows path handed to that shell — C:\Users\...\seqeditor.cmd — gets every backslash silently eaten by ordinary POSIX word-splitting before it's treated as a command at all. The error that came back, C:UsersitaisAppData...seqeditor.cmd: command not found, is that mangled path, verbatim.
  3. Even correctly quoted, a .cmd file still isn't something sh can execute directly — it isn't a native binary and sh doesn't consult Windows' file-association rules the way cmd.exe does.
🔧

The fix

Drop the batch file entirely. GIT_SEQUENCE_EDITOR is now set to a plain cp invocation — a POSIX tool Git for Windows already bundles for exactly this kind of hook — with every path wrapped in single quotes rather than double quotes or left bare: inside single quotes, POSIX shells preserve a backslash literally instead of treating it as an escape character. The same single-quoting was applied to the exec git commit --amend -F line for reworded commits, which goes through the identical shell and had the identical exposure. Verified by rewording and squashing commits in a live rebase and watching the resulting history — new message applied, commit count reduced by exactly the squashed commit, nothing left behind in the working tree.

Also fixed along the way: the per-commit action control was a plain WPF ComboBox, which renders with its default white background and pale gray text regardless of the app's dark theme — there was no existing dark ComboBox style anywhere in the app to reach for. Rather than write one, the control became a themed button that cycles through the five actions on click, matching every other toggle in this update instead of introducing a new one.
04

Hardening

All three bugs above were found the same way every bug in this project has been: build it, then actually run it against a disposable repository and look at what's on screen — not just read the diff and trust that it compiles.

✅

What live testing actually caught this round

A scratch repository with eight commits — a C# file edited twice, a PNG recolored, three small commits built specifically to squash and reword — surfaced all three bugs above on the very first pass through each feature. None of them were visible from the source diff alone: the image-diff inversion only shows up once you're looking at two loaded images with contradictory captions; the tab-clipping only shows up in a rendered window; and the sequence-editor bug only shows up the moment git itself tries to invoke it, since C# code that builds a correct-looking string has no way to know which shell will eventually parse it.

Every item deferred out of Phase 03 and Phase 04 is closed. One thing from the original plan remains open, untouched by this round.

Methodology. Same as every phase before it: built, then exercised live against a disposable scratch repository, driven through the real UI via Windows automation, and checked against screenshots rather than trusted from the source. All three bugs above were found this way, in the first pass through each feature, not in code review.

The user's real repositories were never opened during this round; settings were backed up before the first test launch and restored immediately after the last one.
