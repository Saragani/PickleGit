PickleGit — Engineering Changelog

Building PickleGit

A phase-by-phase record of turning a working WPF git client into a Git daily driver — conflict resolution, hunk staging, rebase, a command palette, hosted pull-request lookups, and a fair number of things that broke along the way and got fixed before they shipped.

PHASE 00

Foundations

Before any feature work, the app needed plumbing that wouldn't buckle under everything built on top of it: a threading model that couldn't deadlock, a CLI escape hatch for what LibGit2Sharp refuses to do, and a way to notice changes without hammering the disk every two minutes.

  • GitExecutor — a single dedicated thread now owns every LibGit2Sharp call. Nothing touches the Repository object from the UI thread or a bare Task.Run again; every later feature routes through it.
  • Single-walk ref-mask visibility — branch membership is computed as a bitmask during the one history walk that already runs, removing a second full walk that existed purely to answer "is this commit on my branch."
  • Paged history + skip-unchanged refresh — a "Load more" footer replaces the hard commit cap, and a state-signature check skips rebuilding the graph and re-serializing the cache when nothing has actually changed.
  • RepositoryWatcher — a debounced FileSystemWatcher on the working directory and .git replaces a 2-minute polling timer with near-instant external-change detection. The timer survives only as a 5-minute failsafe for network drives.
  • GitCli + CliGitService — the process runner and typed wrapper around git.exe that every later CLI-backed feature — rebase, SSH, GPG, LFS — is built on top of.
  • Dialog infrastructure — DialogService with themed Prompt / Confirm / Error windows, retiring Microsoft.VisualBasic.InputBox and raw MessageBox calls for good.
  • Keyboard shortcut baseline — F5, Ctrl+O, Ctrl+W, Ctrl+Tab, Ctrl+Enter, and Ctrl+F reserved for search — the skeleton every later shortcut hangs off.
  • Cleanup — annotated tags stopped signing with a hardcoded user / user@example.com; an empty MainViewModel and a stray #if false block were deleted.
PHASE 01

Core Operations

With the plumbing in place, this phase filled in the operations a daily-driver git client is expected to have and PickleGit simply didn't: amend, force-push, real stash management, and context menus in every place a right-click came up empty.

  • Commit polish — an amend checkbox that prefills the HEAD message, a sign-off toggle, and a split Commit & Push button.
  • Push options — automatic upstream tracking for new branches, force-with-lease behind a danger confirmation, and pushing individual tags.
  • Pull & fetch options — a pull-rebase dropdown next to the plain pull button, plus fetch --prune and fetch-all-remotes.
  • Branch completeness — checkout a remote branch as a new local tracking branch, rename branches, a fixed delete-force flag that was previously ignored, and a detached-HEAD badge for detached checkouts.
  • Reset & revert — soft, mixed, and hard reset plus revert, added to the commit context menu.
  • Stash, actually finished — apply, pop, and drop from a context menu, plus a diff preview on selection — PopStash existed on the view model before this but had no way to reach it from the UI.
  • Tags & remotes — delete / push / checkout a tag; add, edit, and remove a remote — previously list-only.
  • File context menus — open, reveal in Explorer, copy path, add to .gitignore, and discard, on both staged and unstaged files.
  • Commit context menu — copy SHA, copy message, checkout detached, revert, reset here, and rebase onto (wired up fully in Phase 04).
  • Commit search — a Ctrl+F filter bar over message, author, and SHA that narrows the graph live.
  • Tab context menu — close, close others, copy repository path, and open in Explorer or a terminal.
PHASE 02

Conflicts & Undo

The trust threshold. A client that can't show you a conflict, or take back a merge gone wrong, isn't one you reach for on a bad day. This phase made both of those things visible and reversible.

  • Conflict detection — ConflictState reads .git/MERGE_HEAD, CHERRY_PICK_HEAD, and the rebase-merge directory directly; merge, cherry-pick, and revert stopped throwing on conflict and return a real result instead.
  • Conflict banner — a working-directory banner listing every conflicted file with Take Ours / Take Theirs / Mark Resolved, plus Continue and Abort for the operation as a whole.
  • Three-pane merge editor — a diff3-aware marker parser feeds a dedicated MergeConflictEditorWindow — Ours and Theirs read-only side by side, an editable Result pane below, per-block accept buttons.
  • Undo — UndoService records the pre- and post-operation HEAD around every mutating action and can reverse a commit, a hard HEAD move, a checkout, or a branch delete from a single status-bar button.
  • rerere — an opt-in setting that turns on git's own conflict-resolution memory.
🐞

Caught in testing

The conflict-marker parser only captured the marker lines themselves (<<<<<<<, =======, >>>>>>>) — the actual conflicting content in between was silently dropped. Fixed by appending every line, not just delimiters, to the raw buffer.

A binding on <Run Text="{Binding UnresolvedCount}"/> defaulted to TwoWay against a getter-only property and crashed the whole process on open — WPF has no global safety net for a bad binding. Fixed the binding to OneWay, and added an Application.DispatcherUnhandledException handler as a backstop so a single bad binding can never take the app down again.

PHASE 03

Diff & Staging

Staging a whole file at a time is a blunt instrument. This phase brought staging down to the hunk and the line, and made the diff panel itself worth looking at.

  • Hunk staging — Stage / Discard / Unstage buttons on every hunk header, built on PatchBuilder constructing real unified-diff text piped into git apply --cached.
  • Line staging — a checkbox gutter on added and removed lines with Stage / Discard / Unstage Selected — PatchBuilder converts unselected deletions back to context and drops unselected additions before building the patch.
  • Word-level diff — an LCS pass over word tokens on paired remove/add lines highlights the exact span that changed, not just the whole line.
  • Diff navigation — F7 / Shift+F7 jump between hunks without leaving the keyboard.
  • Staged and unstaged lists, finally wired up — the file lists in the commit-detail panel existed but selecting a row did nothing; they now drive the diff panel like everywhere else in the app.
🐞

Caught in testing — the hard one

Hunk and line staging on a file's last hunk consistently failed with error: patch does not apply, no matter how carefully the patch was reconstructed by hand. Three separate manual reconstructions with git apply --check all succeeded, which ruled out the obvious suspects.

The actual cause: ParsePatch split the raw patch text on '\n', and a string ending in a newline produces a trailing empty-string element from Split() that isn't real patch data. Left uncorrected, that phantom element became a bogus zero-length context line on the last hunk, corrupting its declared line count. Fixed by detecting and dropping the trailing empty element before parsing.

Deferred: side-by-side diff view, syntax highlighting, image diff for binary assets, and file history & blame were scoped for this phase but not built — they remain open.
PHASE 04

Rebase & Drag-Drop

Grab a branch, drop it somewhere, decide what that meant — needed rebase, a reflog to fall back on, and drag-drop plumbing that had no precedent anywhere else in the app.

  • Rebase onto — from a branch or commit context menu, run via CLI with --autostash; a conflict mid-rebase flows straight into the Phase 02 banner instead of a dead end.
  • Reflog — parsed directly from .git/logs/HEAD (LibGit2Sharp's own reflog API proved too weak) into a hidden-by-default sidebar section with checkout and hard-reset actions — the safety net standing behind Undo.
  • Drag-drop branch operations — dragging a branch onto another opens a merge/rebase menu; dragging a commit onto a branch cherry-picks it. Built on WPF's native DragDrop.DoDragDrop, a different mechanism from the custom tab-drag ghost used elsewhere in the shell.
🐞

Caught in testing

git pull --rebase stopping for a conflict is normal, expected behavior — but it exits non-zero, and the generic CLI-error path treated any non-zero exit as failure and popped a scary "Operation Failed" dialog instead of the conflict banner. Fixed by adding RunCliAllowingConflictAsync, which checks whether the repo is genuinely mid-conflict before deciding a non-zero exit is a real failure, and switching both pull-rebase and the new rebase-onto command to use it.

✅

Cross-phase proof

Cherry-picking a commit by drag-drop onto a branch that had diverged produced a genuine merge conflict on a shared line — and it resolved cleanly through the Phase 02 banner (Take Theirs, Continue). Two phases built weeks apart, working together on the first real test.

Deferred: the interactive rebase dialog — reorder, squash, and reword commits before a rebase — was scoped as an XL item and pushed out rather than rushed.
PHASE 05

UX Platform

The features under the hood were catching up to a real client; the surface around them still had no settings window, no way to search for a command, and no way to push over SSH. This phase closed all three, plus GPG signing and a commit-message guide most git clients take for granted.

  • SSH remotes — LibGit2Sharp 0.27's credential handler can't drive SSH auth at all. Fetch, pull, push, push-tag, and clone now detect ssh:// / git@ URLs and route through the CLI, letting the system's own OpenSSH client and agent do the work.
  • GPG signing — a Settings toggle, or an existing commit.gpgsign config, routes the commit through git commit -S via the CLI instead of the unsigned LibGit2Sharp path.
  • Settings window — General (git.exe path override with a browse button, commit-history load limit) and Git Behavior (rerere, GPG signing) — the app's first dedicated settings surface, reached from a new toolbar gear button.
  • Command palette — Ctrl+P opens a fuzzy-searchable list of app and per-repo commands in a borderless popup — Ctrl+Shift+P was already taken by Push, so the palette took the other conventional slot instead.
  • Commit message guides — a live subject-line counter that turns red past 50 characters, a wrap hint for the body, and auto-fill from the repo's commit.template the first time the staging view opens.
🐞

Caught in testing

Choosing a palette command that opens its own dialog — Create Branch, for instance — crashed with "Cannot call Show, ShowDialog, Close… while a Window is closing." The palette closed itself and executed the chosen command in the same call stack, but WPF forbids opening a new window while another is still tearing down. A second, subtler issue compounded it: closing the palette shifted focus, which re-fired its own Deactivated handler and called Close() a second time on a window already mid-close.

Fixed by having the palette record the chosen command and close itself with nothing else running, then letting the caller read that choice and execute it only after ShowDialog() has fully returned — plus a simple re-entrancy guard so Close() can never fire twice.

PHASE 06

Hosting & Ecosystem

The last phase reached outside the local repository entirely: pull requests from GitHub, GitLab, and Bitbucket Cloud in the sidebar, avatars on the graph, and the parts of a repo that live beside it rather than inside it — submodules, worktrees, LFS.

  • Hosting provider abstraction — one IHostingProvider interface with GitHub, GitLab, and Bitbucket Cloud implementations, auto-detected by parsing the origin remote's domain, owner, and repo slug.
  • Pull request sidebar — a live list of open PRs with title, source → target route, and author; double-click opens it in the browser. Works unauthenticated for public repositories, and picks up a stored token automatically for private ones.
  • Settings → Integrations — add, list, and remove hosting tokens — a personal access token for GitHub/GitLab, a username plus app password for Bitbucket — stored through the existing Windows Credential Manager wrapper.
  • Avatars — AvatarService resolves GitHub no-reply addresses directly to a GitHub avatar and falls back to Gravatar for everything else, disk-caches the result, and draws it as a circle on the commit graph node — initials remain the fallback when no avatar exists.
  • Submodules sidebar — lists every submodule with a clean / modified / not-initialized status, with init, update, and open-as-a-new-tab actions.
  • Worktrees sidebar — list, add (from a new or an existing branch), remove, and open any worktree as its own tab.
  • Git LFS awareness — a diff for an LFS-tracked file shows a plain-language banner instead of a meaningless pointer-file diff, plus "Track with Git LFS…" and "Install Git LFS" as direct passthroughs to the real git lfs commands.
🐞

Caught in testing — three, this time

Credential storage. Hosting tokens were keyed as host:<domain>, but the existing credential store splits its target string on a single colon and takes the first two parts — the extra colon in host: shifted every field over by one, so a saved token could never be found again. Fixed by moving the marker into the username slot instead of the host, where an extra colon can't corrupt the split.

Adding a worktree. Creating a worktree with a brand-new branch name failed with fatal: invalid reference — git worktree add only checks out an existing ref unless told otherwise. Fixed by checking the typed name against the local branch list and adding -b only when it doesn't already exist.

Removing a worktree. Removing a worktree whose directory was still open in another tab left that tab throwing reference 'HEAD' not found on every refresh, since its working directory had just been deleted out from under it. Fixed by checking whether the repo path still exists before any refresh attempt, and quietly stopping that tab's watcher and auto-refresh timer the moment it doesn't — no more repeated error dialogs for a tab that can't recover.

✅

Verified without touching a real token

All three providers were changed to attempt an unauthenticated call before falling back — public repositories don't need a token at all. That made it possible to verify the entire pull-request pipeline end to end against the real, public microsoft/vscode repository on GitHub, including a live avatar fetch for Linus Torvalds's GitHub account, without ever touching the user's actual Bitbucket credentials.

Architecture, in short

Four decisions made in Phase 00 held up across every phase after it without needing to be revisited.

Hybrid backend

GitService (LibGit2Sharp) handles reads, status, and index operations. GitService.Cli takes over for anything LibGit2Sharp can't do — rebase, SSH, GPG, LFS, patch application — and the libgit2 handle is reopened after every CLI mutation.

GitExecutor

One dedicated background thread. Every libgit2 and CLI call funnels through it, which serializes the two backends against each other for free and keeps the Repository object off the UI thread permanently.

RepositoryWatcher

Debounced FileSystemWatchers on the working directory and .git, with app-initiated operations wrapped in a suppression scope so PickleGit's own writes don't trigger a refresh of themselves.

Dialogs & MVVM

Every prompt, confirmation, and error goes through DialogService's themed windows. No code-behind logic outside established patterns; every view stays bound to a view model.

What's left

Scoped, not forgotten — the items pushed out of a phase rather than cut from the plan.

  • XLInteractive rebase — reorder, squash, and reword commits before a rebase runs. Deferred out of Phase 04.
  • LSide-by-side diff, syntax highlighting, image diff, and file history & blame — scoped for Phase 03, not yet built.
  • P7Provider-native PR creation and richer hosting integration beyond browser hand-off — the natural next step after Phase 06's read-only sidebar.

Methodology. Every feature in this log was built, then exercised live — against disposable scratch repositories, driven through the real UI via Windows automation and verified with screenshots, not just read back from the source. The eight bugs above were all found this way, not in code review.

The user's own repositories and settings were backed up before any test that touched them and restored immediately after — nothing in this log was verified at the cost of real work.
