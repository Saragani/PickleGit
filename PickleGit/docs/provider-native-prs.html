PickleGit — Engineering Changelog · Hosting

Provider-Native PRs

Every prior changelog in this project flagged the same gap: PickleGit could only ever open a browser to start a pull request, never actually create one. This round closes it — PickleGit now calls GitHub, GitLab, and Bitbucket's own APIs to create the PR directly, with a new dialog to pick the title, description, target branch, and draft state, and three places in the UI that can trigger it.

01

What Shipped

IHostingProvider previously had one PR-creation method: GetCreatePrUrl, a string builder for a browser compose page. It never called an API — "create" meant "open a link and let the user do the rest." That's still there as a fallback, but it's no longer the only path.

  • Real API creation — a new CreatePullRequestAsync(title, description, source, target, draft) on IHostingProvider, implemented against each provider's actual REST endpoint: POST /repos/{o}/{r}/pulls on GitHub, POST /projects/{id}/merge_requests on GitLab, POST /repositories/{o}/{r}/pullrequests on Bitbucket Cloud.
  • A real dialog, not a raw URL — Views/Dialogs/CreatePullRequestDialog.xaml lets you set the title, an optional description, pick the target branch from a dropdown of actual remote branches, and toggle draft — instead of guessing what query parameters a browser page will accept.
  • Graceful fallback, not a hard requirement — if no token is stored for the host (HostingProvider.IsConfigured == false), PickleGit still opens the browser compose page — now pre-filled with the title and description you just typed, via extended GetCreatePrUrl query parameters, instead of a bare branch comparison.
  • A PR badge on the branch that owns one — after the PR list loads, every local and remote branch tree node is checked against open PR source branches; a small PR pill appears next to any branch that already has one open, so you don't fire off a duplicate.
✅

Why fallback stays instead of disappearing

Requiring a token before "Create Pull Request" would work at all turns a nice-to-have (skip the browser) into a blocker (can't create a PR without first visiting Settings → Integrations). The command still works with zero configuration — it just does less for you until you add a token.

02

Three Ways In

The dialog is one shared code path (ShowCreatePullRequestDialog), but there are now three ways to reach it, matching the three places you'd naturally think "I want to open a PR from here."

🌿

Branch context menu

Right-click any local branch → Create Pull Request…. Source is that branch; target defaults to main/master if either exists on the remote.

🖱️

Drag one branch onto another

The existing drag-drop popup (Merge / Rebase) now offers a third option — Create Pull Request 'source' → 'target'… — with both branches already chosen from where you dropped.

📋

Pull Requests sidebar section

The pre-existing Create Pull Request… item there is unchanged in name, but now opens the same dialog instead of guessing a target and jumping straight to a browser.

Mechanics

Auth reuses exactly what GetPullRequestsAsync already loaded from HostingCredentials — no new credential path, no new place a token is written to disk. The three providers disagree on a few details, so each got its own small adapter.

ProviderAuth headerCreate endpointDraft handling
GitHub Authorization: token … POST /repos/{o}/{r}/pulls native "draft": true field
GitLab PRIVATE-TOKEN: … POST /projects/{id}/merge_requests no API field — prefixes the title with "Draft: ", same as the GitLab UI does
Bitbucket Cloud Authorization: Basic … POST /repositories/{o}/{r}/pullrequests not supported by Bitbucket Cloud — checkbox hidden in the dialog for this provider

Error handling

  1. A non-2xx response body is parsed for that provider's real error shape — GitHub's {"message", "errors": [...]}, GitLab's {"message"}, Bitbucket's {"error": {"message"}} — and surfaced through DialogService.ShowError, not a generic "request failed."
  2. On success, the newly created PR's real web URL (from the API response, not a guessed one) opens in the browser immediately, and LoadPullRequestsAsync() re-runs so the sidebar list and the new branch badge both pick it up without a manual refresh.

What stayed the same on purpose

  • Self-hosted GitHub Enterprise / GitLab / Bitbucket Server are still out of scope — HostingProviderFactory still matches only the three .com/.org domains it always did. Native creation inherits that same limit rather than trying to solve it.
  • The credential UI in Settings → Integrations is untouched — one token per domain was already enough to authenticate creation, not just listing.

What's left now

The one item every prior changelog flagged as outside the original plan's scope is now inside it.

  • DONEProvider-native PR creation — GitHub, GitLab, and Bitbucket Cloud all create through their own API, with a browser fallback when no token is configured.
  • DONERicher hosting integration — branch-level PR badges, a drag-drop entry point, and a real title/description/draft dialog replace the old "guess a target and open a link" flow.
  • laterSelf-hosted instances — GitHub Enterprise, self-managed GitLab, Bitbucket Server — would need a user-configurable domain-to-provider mapping; not attempted here.
  • laterReviewers, labels, and existing-PR edits — creation covers the common case; assigning reviewers or updating a PR already open are natural next steps if they turn out to matter in practice.
